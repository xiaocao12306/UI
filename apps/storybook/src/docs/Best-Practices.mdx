## API Design

- Keep props minimal and composable; avoid business-specific flags.
- Prefer controlled APIs for overlays (`open`, `onOpenChange`).
- Expose semantic variants (`solid`, `outline`) instead of hardcoded colors.

## Theming

- Do not hardcode color values inside component internals.
- Resolve styling via Aurora CSS variables only.
- Keep layering (`z-index`) tokenized (`--aurora-z-overlay` / `--aurora-z-modal` / `--aurora-z-toast`), not magic numbers.
- Validate stories under all four theme packs before merge.

## Accessibility

- Ensure keyboard dismissal paths for overlay components.
- Keep visible labels or accessible names for form controls.
- For icon-only buttons, provide non-empty `aria-label` or `aria-labelledby` on the button itself; blank values are invalid, `aria-labelledby` stays canonical when both are present, inline child `aria-label` is treated as fallback only, and decorative glyph text should remain `aria-hidden`.
- Keep button pressed-state feedback scoped to unmodified activation keys (`Enter`/`Space`, including `Space` and legacy `Spacebar` key values); ignore `Ctrl/Meta/Alt` chords and IME composition key paths (`isComposing` / `keyCode=229`) so host shortcuts and CJK confirmation do not flash pressed visuals.
- When app-level shortcuts preempt button activation keydown with `preventDefault()`, keep button pressed feedback side-effect free.
- Keep button keyboard-intent tracking bound to each control `ownerDocument` so focus-visible fallback still appears when users Tab in from adjacent controls, including iframe-hosted renders.
- Respect OS reduced-motion preferences: keep `GlobalStyles` mounted so surfaces tagged with `data-aurora-reduced-motion` disable shimmer/spin/indeterminate motion (Button loading spinner, Progress indeterminate bar, Skeleton shimmer) under `prefers-reduced-motion`, and keep timer-driven feedback (`LoadingDots`, `StreamingText`) on default `respectReducedMotion` behavior unless the product explicitly requires animated narration.
- For progress bars, prefer `ariaLabelledBy` when a visible heading already exists; use `ariaLabel` (or `label` fallback) for standalone bars without nearby title context.
- For loading dots status narration, prefer `ariaLabelledBy` when a visible heading already exists; use `ariaLabel` (or `label` fallback) for standalone loading indicators.
- For streaming status narration, prefer `ariaLabelledBy` when a visible heading already exists; use `ariaLabel` (or `label` fallback) for standalone streaming regions.
- For narrated skeleton states, prefer `ariaLabelledBy` when a visible loading heading already exists; use `ariaLabel` for standalone placeholders without nearby heading context.
- Prefer `FormField` wrappers for helper/error copy so `aria-describedby` and `aria-errormessage` stay in sync.
- For composite/non-clonable field controls, provide `htmlFor` on `FormField` and a matching control `id`; otherwise label association cannot be inferred.
- If a wrapped control already provides `aria-labelledby`, avoid passing `aria-label` together; keep `aria-labelledby` as the canonical name source.
- For icon/non-text FormField labels, provide control-level `aria-labelledby` when a visible heading exists, otherwise provide `aria-label`; rich label nodes with inline `aria-label` are auto-detected, and development builds warn when control naming is still missing.
- Treat blank input `aria-label`/`aria-labelledby` as invalid config and remove them; keep `aria-labelledby` as the canonical source when both naming props are present.
- Treat blank checkbox `aria-label`/`aria-labelledby` as invalid config and remove them; keep `aria-labelledby` as the canonical source when both naming props are present.
- Keep checkbox shortcut hints precise: expose `aria-keyshortcuts="Space"` only while actionable, and remove shortcut metadata when checkbox is disabled.
- For icon/non-text checkbox label or description content, provide `aria-labelledby` when a visible heading exists, otherwise provide `aria-label`; rich content with inline `aria-label` is auto-detected, and development builds warn when naming is still missing.
- Keep select shortcut hints precise: expose `aria-keyshortcuts="ArrowDown ArrowUp"` only while actionable, and remove shortcut metadata when select is disabled.
- Keep native select `<option value>` entries unique and stable; development builds warn on duplicate values so selected-option semantics stay deterministic.

- Treat blank date-picker `aria-label`/`aria-labelledby` as invalid config and remove them; keep `aria-labelledby` as the canonical source when both naming props are present.
- Keep switch keyboard toggle and pressed-state feedback scoped to unmodified `Space` keys (` ` / `Space` / `Spacebar`) and ignore `event.repeat` long-press pulses; `Ctrl/Meta/Alt + Space` should remain host shortcut behavior, and IME composition key paths (`isComposing` / `keyCode=229`) should stay no-op.

- Keep combobox keyboard-intent tracking bound to each control `ownerDocument` so focus-visible fallback still appears when users Tab in from adjacent controls, including iframe-hosted renders.
- For guarded approval flows, use control-level `onClick(event.preventDefault())` on switch-like toggles to block pointer state mutation while preserving keyboard/readout semantics.
- When using standalone DatePicker error messaging, pass `errorMessageId` with `invalid` so both `aria-describedby` and `aria-errormessage` stay aligned.
- Preserve nuanced `aria-invalid` values (`grammar`/`spelling`) when integrating validation services; avoid normalizing them to generic `true`.
- For tooltip-enhanced controls that already have helper text, keep `aria-describedby` lifecycle clean: append tooltip ID only while visible and remove it on close.

- For icon/non-text combobox options, provide at least one searchable metadata path (`options[].textValue`, `options[].ariaLabel`, or `keywords`) so filtering and selected-input echo stay deterministic even when visual labels are glyph-only.
- For command-palette `aria-activedescendant` flows, keep `aria-selected` on the active option so screen readers announce current keyboard focus target.
- For `aria-activedescendant` combobox flows, keep listbox options out of tab order and preserve input focus during pointer selection.

## Overlay Policies

- Dialog/Drawer should explicitly define dismiss policy (`closeOnEscape`, `closeOnOutsidePointer`) for critical workflows.
- Keep `restoreFocus` enabled for dialogs unless product flow intentionally redirects focus after close.
- Keep `restoreFocus` enabled for drawers unless closing should intentionally hand focus to a different target.
- Prefer `ariaLabelledBy` for dialog/drawer naming when a visible heading already exists; use `ariaLabel` only for standalone/non-text titles without nearby heading context.
- For icon/non-text dialog/drawer titles, provide `ariaLabelledBy` when a visible heading exists, otherwise provide `ariaLabel`; development builds warn when both naming paths are missing.
- Keep close-button focus intent primary-only in dialog/drawer surfaces so non-primary pointer interactions (for example context/right-click) do not clear keyboard focus-visible feedback.
- Keep dialog/drawer close-button keyboard pressed-state feedback scoped to unmodified activation keys (`Enter`/`Space`, including `Space` and legacy `Spacebar` key values); keep `aria-keyshortcuts="Enter Space"` visible for AT/QA discoverability, and ensure preempted keydown (`event.defaultPrevented`), `Ctrl/Meta/Alt` chords, plus IME composition key paths (`isComposing` / `keyCode=229`) do not trigger pressed visuals.
- Keep dialog/drawer Escape shortcut hints explicit (`aria-keyshortcuts="Escape"`) only when `closeOnEscape` is enabled so dismiss affordance metadata matches runtime policy.
- Popover and Dropdown should restore focus to trigger on keyboard dismissal (Escape), but keep pointer target focus on outside-click dismissal.
- For blocking confirmation/action menus, set `closeOnEscape={false}` + `closeOnOutsidePointer={false}` and provide an explicit close/confirm path.
- For conditional runtime approval guards, keep `closeOnEscape/closeOnOutsidePointer` enabled and intercept `onEscapeKeyDown/onPointerDownOutside` with `event.preventDefault()` only while guard conditions are active.
- Dropdown menus should expose an accessible name via trigger-linked `aria-labelledby`.
- Keep `Dropdown.items[].key` unique and stable; development builds warn on duplicate keys to prevent focus and close-telemetry ambiguity, and runtime auto-suffixes duplicate render keys by item index to avoid React remount churn.
- For nested overlays, dismiss policy should resolve top-layer first so one Escape/outside click does not collapse the full stack.
- For nested popovers, keep top-layer-first dismiss semantics explicit: first Escape/outside interaction closes inner popover and preserves the outer layer until a second dismiss action.
- For overlays built on `DismissableLayer`, ignore Escape while IME composition is active, including legacy browsers that only expose `keyCode=229`, so CJK text confirmation does not close parent surfaces unexpectedly.
- For overlays built on `DismissableLayer`, keep `Ctrl/Meta/Alt + Escape` on host/app shortcut behavior; modified Escape should not trigger dismiss hooks or close transitions.
- For overlays built on `DismissableLayer`, treat repeated Escape keydown (`event.repeat`) as no-op so long-press does not close multiple overlay layers or duplicate close telemetry.
- Keep tooltip Escape close behavior scoped to unmodified, non-repeat, non-IME keydown so host shortcut chords and CJK composition confirmation cannot collapse hint overlays unexpectedly.
- For persistent hint workflows, set tooltip `closeOnEscape={false}` and rely on blur/pointer-leave close paths; ensure shortcut hints do not expose `aria-keyshortcuts="Escape"` when Escape dismiss is disabled.
- For dialog/drawer in iframe or multi-root hosts, rely on owner-document portal mounting (`ownerDocument.body`) so overlay content and scroll-lock scope stay inside the invoking document.
- For popover/dropdown in iframe or multi-root hosts, validate Escape/outside-pointer dismissal against the overlay `ownerDocument`; cross-document events and `Ctrl + primary` context-click pointer paths should not close the active surface.
- For dialog/drawer flows under app-level shortcuts, rely on built-in Escape preemption behavior so `onEscapeKeyDown` hooks stay side-effect free when Escape is already prevented.
- Keep trigger `aria-controls` conditional on open state for popover/menu patterns to avoid stale ID references.
- Use dialog `onCloseReason` for deterministic telemetry (`close-button` / `escape-key` / `outside-pointer`) instead of inferring dismiss origin from external state transitions.
- Use drawer `onCloseReason` for deterministic telemetry (`close-button` / `escape-key` / `outside-pointer`) instead of inferring dismiss origin from external state transitions.
- Keep dialog/drawer close callbacks ordered and single-purpose: dismiss paths should emit `onCloseReason` before `onOpenChange(false)` to avoid telemetry/state reducer race conditions.
- Keep dialog/drawer handled dismiss events preempted (`event.preventDefault()` on accepted Escape/outside-pointer paths) so downstream document-level shortcuts can detect `defaultPrevented` and skip duplicate close side effects.
- For dropdown typeahead, set `textValue` when item labels are non-string or need locale/accent-friendly buffered matching aliases.
- Ignore dropdown typeahead key matching during IME composition, including legacy browsers that only expose `keyCode=229`, so CJK input confirmation does not shift active menu focus unexpectedly.
- When app-level shortcuts preempt Escape with `preventDefault()`, keep dropdown-level Escape hooks side-effect free and rely on built-in preemption skip behavior.
- When app-level shortcuts preempt Escape with `preventDefault()`, keep popover-level Escape hooks side-effect free and rely on built-in preemption skip behavior.
- Use popover `onCloseReason` for deterministic telemetry (`trigger-click` / `escape-key` / `outside-pointer` / `tab-key`) instead of inferring dismiss origin from focus and pointer side effects.
- Use dropdown `onCloseReason` for deterministic telemetry (`trigger-click` / `item-select` / `escape-key` / `outside-pointer` / `tab-key`) instead of inferring close origin from focus/state side effects.
- Keep popover/dropdown close callbacks ordered and single-purpose: close paths should emit `onCloseReason` before `onOpenChange(false)`, while dropdown `item-select` should run `onSelect` before close callbacks.
- Keep popover trigger keyboard-open behavior scoped to unmodified `ArrowDown`; `Ctrl/Meta/Alt + ArrowDown` should remain available for host/browser shortcut handling.
- When app-level shortcuts preempt popover `ArrowDown`/`Tab` keydown with `preventDefault()`, keep trigger open and boundary Tab-dismiss handlers side-effect free.
- Keep popover trigger shortcut hints explicit (`aria-keyshortcuts="ArrowDown"`) only while closed so keyboard-open affordance stays discoverable without advertising unavailable shortcuts once open.
- Keep popover boundary Tab behavior aligned with document order: keep popover open while focus stays inside, then close on first/last boundary Tab and continue focus to adjacent controls; when no adjacent control exists, fall back to trigger focus.
- Keep popover dialog shortcut hints explicit (`aria-keyshortcuts="Tab [Escape]"`): always advertise boundary Tab dismiss, and include `Escape` only when `closeOnEscape` is enabled.
- Keep dropdown `Tab` behavior aligned with document order: close menu and continue focus to next control; if no next control exists, fall back to trigger focus.
- Keep dropdown `Shift+Tab` behavior aligned with document order: close menu and continue focus to the previous control; if no previous control exists, fall back to trigger focus.
- For controlled dropdowns, treat Escape/outside/trigger dismiss as close intent only; apply final visual close in parent rerender to avoid state race conditions.
- For icon-only/non-text dropdown items, prefer `items[].ariaLabelledBy` when a visible heading already exists; otherwise provide `items[].ariaLabel`. If explicit naming metadata is omitted, `items[].textValue` is used as accessible-name fallback; development builds warn only when `ariaLabel`/`ariaLabelledBy`/`textValue` are all missing.
- For icon-only/non-text dropdown items, provide `items[].textValue` unless readable label text or `ariaLabel` is already available; buffered typeahead can use readable label text, `ariaLabel`, or `textValue`.
- Keep dropdown menu-item keyboard activation explicit and cross-browser (`Enter` / `Space` / legacy `Spacebar`) so non-pointer flows can execute actions without click simulation.
- Keep dropdown navigation/activation/dismiss scoped to unmodified keys; `Ctrl/Meta/Alt + Arrow/Home/End/PageUp/PageDown/Enter/Space/Escape` should stay on host/browser shortcut behavior.
- When app-level shortcuts preempt dropdown keydown with `preventDefault()`, keep trigger/menu keyboard handlers side-effect free so host shortcut guards remain authoritative.
- Keep open-menu roving focus stable under modified `ArrowDown/ArrowUp`; `Ctrl/Meta/Alt` arrow chords should not move dropdown active item.
- Keep dropdown trigger shortcut hints explicit (`aria-keyshortcuts="ArrowDown ArrowUp"`) only while closed so keyboard-open affordance stays discoverable without advertising unavailable shortcuts once open.
- Keep dropdown trigger open targeting stable: `ArrowDown` should focus the first enabled item, and `ArrowUp` should focus the last enabled item for immediate tail-action access.
- When all dropdown items are disabled, keep menu-open focus anchored on the menu surface (`tabIndex=-1`) so keyboard and assistive users can still discover disabled workflow context.
- Keep dropdown menu shortcut hints state-aware (`aria-keyshortcuts="[ArrowDown ArrowUp Home End PageDown PageUp] Tab [Escape]"`): always expose `Tab`, expose `Escape` only when `closeOnEscape` is enabled, and omit navigation hints when at most one item is actionable so QA/AT metadata does not advertise unavailable movement paths.
- Keep dropdown Arrow/Home/End/Page handlers aligned with actionable count: when at most one item is actionable, do not consume navigation keys so host/container keyboard behavior is not preempted by no-op menu movement.
- For long dropdown menus, rely on `PageUp` / `PageDown` jumps to move across actionable items in larger steps while still skipping disabled entries.
- Treat dropdown menu-item activation repeat (`event.repeat`) as no-op after first keydown to prevent duplicate command execution and telemetry jitter during long-press.
- Deduplicate synthesized keyboard-origin click (`detail=0`) after handled dropdown item activation keydown so one keyboard action maps to one item side effect and one close telemetry event.
- Expose dropdown menu-item `aria-keyshortcuts="Enter Space"` only on actionable items; disabled items should omit shortcut hints to avoid false affordance.
- For icon-only/non-text popover/dropdown triggers, prefer `triggerAriaLabelledBy` when a visible heading/label already exists; otherwise provide non-empty `triggerAriaLabel` for standalone controls. Rich trigger labels with inline `aria-label` are auto-detected, and development builds warn when both naming paths are missing.
- Keep field/button focus intent primary-only so non-primary pointer interactions and `Ctrl + primary click` (for example macOS context click) do not clear keyboard-origin focus-visible rings before actual activation intent.
- Keep dialog/drawer close-button focus intent owner-document aware so keyboard re-entry (Tab from neighboring controls, including iframe hosts) restores focus-visible fallback when `:focus-visible` matching is unavailable.
- Keep dialog/drawer close-button focus intent fallback scoped to unmodified owner-document keydown only; ignore `Ctrl/Meta/Alt + key` chords so host shortcuts do not falsely restore keyboard focus-visible rings.
- Keep dialog/drawer close-button owner-document pointer intent primary-only; non-primary pointer buttons and `Ctrl + primary click` (for example macOS context click) should not clear fallback keyboard focus intent.
- Keep dialog/drawer close-button local pointer down feedback primary-only; non-primary pointer buttons and `Ctrl + primary click` context-click paths should not clear focus-visible state or trigger pressed visuals.
- Keep dialog/drawer close buttons visually state-complete (`hover`/`focus-visible`/`active`) so overlay dismissal affordances match the rest of the component system.
- Command palette and combobox-style overlays should keep text input as the primary focus anchor for navigation.

## AI Interaction Patterns

- Always expose pending/loading states for AI request actions.
- Keep command labels explicit and searchable.
- Keep `commands[].key` unique and stable; development builds warn on duplicate keys to avoid active-option and telemetry drift, and runtime auto-suffixes duplicate option keys by filtered index to prevent React key-collision remount churn.
- Provide `textValue` aliases for non-string command labels and accent-friendly matching terms when command search copy differs from visible naming.
- For icon/non-text command labels, prefer `commands[].ariaLabelledBy` when a visible heading already exists; otherwise provide `commands[].ariaLabel`. Rich labels that expose inline `aria-label` are auto-detected, and `textValue` is now used as an option-name fallback when explicit naming props are absent.
- For non-text command labels, ensure at least one search metadata path is present (`textValue`, `ariaLabel`, or searchable `keywords`); development builds warn when all are missing.
- Provide empty-state guidance when no command matches the query.
- Use `loading` + `loadingContent` during async command fetch so stale options are hidden and command execution/navigation is paused until results are ready.
- Track user query updates with `onQueryChange` for downstream telemetry.
- Treat `onQueryChange("")` on palette close as the telemetry reset signal so external state mirrors the internal query lifecycle.
- Keep live-region narration aligned with actionable commands (enabled matches), not just visible results.
- Keep disabled command entries non-executable and keep palette state unchanged on disabled clicks.
- When filtered results are disabled-only, show explicit helper guidance (for example `disabledResultsMessage`) so users understand why Enter/navigation execution is unavailable.
- Keep commands without `onSelect` inert: selecting them should not close the palette or emit `item-select` telemetry; provide `onSelect` for every executable command entry.
- Keep command-palette `aria-keyshortcuts` aligned with actionable state: omit `Enter` when filtered results contain no enabled command, and omit Arrow/Home/End/Page navigation hints when at most one command is actionable.
- Keep command-palette close-button shortcut hints explicit (`aria-keyshortcuts="Enter Space"`) so keyboard dismiss affordance remains discoverable in AT/QA flows.
- Keep command-palette dialog Escape shortcut hints explicit (`aria-keyshortcuts="Escape"`) only when `closeOnEscape` is enabled so dismiss hint metadata matches runtime policy.
- For batched command execution, set `closeOnSelect={false}` so users can run multiple actions without reopening the palette.
- Treat repeated `Enter` keydown (`event.repeat`) as no-op for command execution so long-press does not trigger duplicate actions or telemetry noise.
- Treat repeated `Escape` keydown (`event.repeat`) as no-op for both query-clear and dismiss paths so long-press does not clear/close the palette unexpectedly.
- Keep option-level activation (`Enter`/`Space`) scoped to unmodified, non-repeat keydown so focused listbox options cannot execute on `Ctrl/Meta/Alt` shortcuts or long-press repeats.
- Keep command-palette option `mousedown` focus guards primary-only but exclude `Ctrl + primary click` context-click paths so pointer guard logic does not swallow host context interactions.
- Expose command-palette option `aria-keyshortcuts="Enter Space"` only on actionable options; disabled options should omit shortcut hints to avoid false affordance metadata.
- Use `PageUp` / `PageDown` for long command sets so keyboard users can jump actionable options in larger steps.
- Keep `Ctrl/Meta/Alt + ArrowDown/ArrowUp/Home/End/Page/Enter/Escape` on native input behavior; do not hijack modified shortcuts for command navigation or dismiss.
- Keep `Shift + Escape` available so command input can follow the standard first-escape-clear / second-escape-dismiss sequence.
- Keep `clearQueryOnEscape` enabled so first Escape clears query intent before full palette dismissal; disable only when product expects immediate close.
- After query updates or command-set refreshes, keep command-palette `aria-activedescendant` valid by pointing only to a currently mounted, enabled option; clear it when no executable option remains, and keep it unset when filtered results are empty.
- For blocking approvals, set `closeOnEscape={false}` and `closeOnOutsidePointer={false}` so command confirmation cannot be dismissed accidentally.
- For conditional approval guards, prefer `onEscapeKeyDown/onPointerDownOutside` + `event.preventDefault()` so dismiss policy can be unlocked at runtime.
- When app-level shortcuts preempt Escape with `preventDefault()`, rely on built-in callback-skip behavior so command-level escape hooks stay side-effect free.
- When app-level shortcuts preempt `Enter` / `Escape` with `preventDefault()`, keep combobox-level command execution and query-clear handlers side-effect free so host shortcut guards stay authoritative.
- For iframe/multi-document hosts, validate command-palette dismiss behavior against the palette `ownerDocument`; Escape/outside events from other documents should not close the active palette.
- When command palette is nested inside another overlay, keep top-layer-first dismiss semantics explicit: first Escape/outside interaction closes the palette layer and leaves parent overlay state unchanged.
- Ignore command list navigation/selection keys while IME composition is active, including legacy browsers that only expose `keyCode=229`, so Enter confirms text composition instead of executing commands.
- During IME composition, treat Escape as text-confirm/cancel only; do not expect query reset or palette dismiss until composition ends.
- Use `onCloseReason` to emit deterministic dismissal telemetry (`close-button` / `item-select` / `escape-key` / `outside-pointer`) instead of inferring close path from side effects.
- Keep callback side effects ordered and single-purpose: `item-select` path should treat `onSelect` as action commit, then `onCloseReason`, then `onOpenChange(false)`; close-button path should use `onCloseReason` before `onOpenChange(false)`.
- Guard Ctrl/Cmd + Enter submission against active IME composition (`isComposing` / `keyCode=229`), repeated keydown (`event.repeat`), and `Alt + Ctrl/Cmd + Enter` host-shortcut chords so long-press or modified shortcuts do not accidentally send prompts.
- Localize prompt input naming and helper copy with `ariaLabel`, `shortcutHint`, and `submittingHint`; expose `aria-keyshortcuts="Control+Enter Meta+Enter"` only while `onSubmit` is provided and not submitting so shortcut metadata does not advertise unavailable submit actions.

- Keep prompt textarea keyboard-intent tracking bound to each control `ownerDocument` so focus-visible fallback still appears when users Tab in from adjacent controls, including iframe-hosted renders.

- Localize message-bubble narration with `speakerLabel` or explicit `ariaLabel`; blank values should be omitted so speaker-based fallback naming remains valid.
- Prefer `ariaLabelledBy` for message bubbles when a visible heading/title already exists; use `ariaLabel` for standalone bubbles without nearby naming context.

- Normalize reasoning step payloads before rendering (or rely on built-in blank-step filtering) so whitespace-only steps do not create empty list bullets; keep `emptyText` non-blank unless default fallback copy is desired.
- Show reasoning disclosure with explicit user intent (default collapsed).
- Use streaming containers that surface busy completion (`aria-busy`) for assistive tech parity.
- Keep streaming-code narration `live="off"` for high-frequency token output; enable `live="polite"` only when progressive code updates must be announced.

- For streaming code regions, prefer `ariaLabelledBy` when a visible heading already exists; use `ariaLabel` (or `label` fallback) for standalone code stream panels.
- Keep `respectReducedMotion` enabled for streaming code by default so reduced-motion users receive immediate full content instead of progressive character animation.
- Keep command-palette close-button keyboard-intent tracking document-scoped so keyboard re-entry (for example `Shift+Tab` from search input) restores focus-visible fallback consistently, including iframe hosts.

## Data Presentation Patterns

- Use typed columns and stable row keys in data tables; fallback source-index keys exist for safety but explicit `rowKey` is preferred.
- Keep `columns[].key` values unique; development builds warn on duplicate column keys, and runtime render keys auto-suffix duplicates by column index to avoid header/cell key-collision remount churn.
- When sortable columns rely on fallback narration (no `sortLabel/sortLabelledBy`), keep `columns[].key` human-readable because fallback sort labels are normalized from key text; opaque/symbol-only keys will degrade to generic `Column N`.
- Keep `rowKey` outputs unique per source row; development builds warn on duplicate keys to catch unstable identity wiring early, and runtime auto-suffixes duplicate outputs by source index to prevent key-collision remount churn.
- If `rowKey` uses `rowIndex`, treat it as source-data index; avoid deriving keys from sorted visual position.
- Keep `rowKey` pure and deterministic: table caches source-row keys and reuses them on sort-only rerenders to avoid key churn and unnecessary key-function re-execution.
- For custom cell renderers, use `render(row, rowIndex, sourceIndex)` and pick the index intentionally (`rowIndex` for ranking UI, `sourceIndex` for source-linked IDs/telemetry).
- Mark the primary identity column with `rowHeader` so screen readers can announce row context consistently.
- Ensure each table has an accessible name through `caption`, `ariaLabel`, or `ariaLabelledBy`; fallback `Data table` naming exists but should be treated as a safety net.
- Expose `aria-sort` only on the actively sorted header so assistive narration reflects current sort state without repeating neutral columns.
- Keep non-sortable headers free of `aria-sort` so assistive tech does not announce inert columns as interactive.
- Validate sortable headers with both pointer and keyboard (`Enter`/`Space`, including legacy `Spacebar`) so non-pointer workflows stay first-class.
- Support sortable-header focus traversal with `ArrowLeft` / `ArrowRight` / `Home` / `End` / `PageUp` / `PageDown` so keyboard users can move between columns without forcing sort changes.
